# Animal Poke · 游戏设计与改进路线图

## 1. 产品重新定义

### 核心承诺

Animal Poke 的核心价值不是“随机给一张动物卡”，而是：

> 玩家在安全、不打扰动物的前提下，通过真实观察获得可信、可解释、可长期收藏的动物记录。

核心循环建议收敛为：

`安全探索 -> 取景质量反馈 -> 可信识别 -> 技巧捕获 -> 可解释生成 -> 收藏/研究 -> 任务/战斗/派遣`

### 北极星指标

上线前不要以页面数或单测数作为完成指标，改用：

- 首次同意后成功完成首捕的比例。
- Detect 到真实收藏完成的端到端成功率。
- 识别 unknown/误识别率与玩家纠正率。
- 单次成功收藏的 P95 延迟、平均 AI 调用数和平均成本。
- D1/D7 留存及首日完成第二次主动探索的比例。
- 数据丢失、重复奖励、资源异常和隐私事故为零。

## 2. 阶段路线

### Phase 0：生产止血（1 周）

目标：停止产生误导、数据风险和错误发布。

- 隐藏或关闭当前假的 VLM 状态、占位 PvP/社交/商业化入口。
- 修复 staging 生产库和 `:dev` 镜像问题。
- 修复前端生产 API 地址。
- 商业化 production 默认关闭。
- 前端测试取消 soft-fail。
- Vision 未独立配置时明确不可用，不回退到 LLM。

退出条件：错误环境不能发布；相机未识别时绝不能进入默认鹅捕获。

### Phase 1：可信最小闭环（2~3 周）

目标：只支持猫/狗/鹅，但一条链路必须真正完成。

- 服务端同意记录与鉴权。
- 单帧手动扫描 MVP，不做 2~5 FPS 连续上传。
- Detect 返回目标，玩家确认后进入捕获。
- 捕获成功后 Analyze -> 确定性 Value -> IDB -> Sync -> Pull。
- 结果页显示物种、置信度、属性依据、同步状态。
- 全链路 E2E、OpenAPI contract、MySQL integration 成为硬门禁。

退出条件：固定 fixture 从相机到服务器收藏可重复通过，刷新与断网恢复不丢数据。

### Phase 2：真正可玩（3~5 周）

目标：把“识别 Demo”变成有技巧、反馈和进程的游戏。

- 投掷力度/方向/物种差异和合法重试。
- 真实图鉴、首捕揭晓、重复捕获价值。
- 商店/签到使用统一经济状态。
- LBS/天气/发现点接入服务端权威范围。
- PvE 战斗接入真实收藏和奖励。
- 首捕教学、错误恢复、移动布局和 WCAG。

退出条件：D1 有完整目标，资源不变量和经济模拟通过，无无限产币或死循环。

### Phase 3：留存与差异化（4~8 周）

目标：建立 D7/D30 价值，而不是堆未接通功能。

- 每日/每周研究任务、城市图鉴目标、派遣。
- 亲密度/研究点/装饰型重复价值。
- 识别纠正与可解释稀有度。
- 账号绑定和跨设备恢复。
- 音效、触觉、低流量/低电量模式。
- 黄金集 nightly 认证和核心漏斗分析。

退出条件：D7 目标可完成，识别质量/成本/SLO 有稳定监控。

### Phase 4：社交、排行和商业化

目标：只有服务端权威、风控、数据和核心留存稳定后再开放。

- 区域排行使用服务端事件和反作弊数据。
- PvP 使用真实 match 状态、战斗日志和 ELO 结算。
- 分享默认保护地点和身份。
- 商业化只销售不破坏公平的便利/装饰内容；平台验签和退款完成后再上线。

退出条件：安全评审、隐私评审、经济压力测试、商店验签和回滚演练通过。

## 3. 人机交互改进

### 3.1 相机与识别

- 权限前软引导，说明“为什么需要”和“拒绝后仍能做什么”。
- 取景框显示光线、清晰度、目标大小和稳定度，不直接声称已识别。
- 状态文案严格对应事实：`准备相机`、`等待扫描`、`上传中`、`识别中`、`找到目标`、`未找到`。
- 多动物时显示可点选框，不替玩家暗中选择。
- unknown 是正常结果，提供重扫和纠正，不回退为鹅。
- 429、离线、后端不可达、Provider 故障必须有不同恢复动作。

### 3.2 捕获

- MVP 使用单手可完成的蓄力/滑动，避免复杂 3D 先拖慢闭环。
- 技巧来自力度、时机和稳定，不来自不可解释的随机数。
- 成功率条展示区间和影响因素，但不泄露可被脚本直接利用的完整风控规则。
- 一次 attempt 的资源、结果和反馈必须原子一致。
- 失败后明确“还可尝试几次、需要什么资源、为什么失败”。

### 3.3 结果揭晓

结果页应按以下顺序：

1. 物种确认和照片缩略图。
2. 稀有度揭晓。
3. 影响因素解释。
4. 新图鉴/重复捕获价值。
5. 同步状态。
6. 下一步：查看图鉴、继续探索或安全结束。

### 3.4 错误体验

- 错误页必须回答：发生了什么、玩家是否损失资源、能否重试、如何重试。
- Provider 失败时不扣最终捕获成本，或明确保留 session 让玩家稍后继续。
- 所有失败显示 request ID 的可复制短码，但不暴露内部错误。
- 不使用“成功 Toast”掩盖未落库、未扣款或未发奖。

## 4. 玩法与进程优化

### 4.1 首日流程

- 第 1 个目标：完成安全首捕。
- 第 2 个目标：在图鉴查看属性依据。
- 第 3 个目标：完成一次无风险 PvE/研究任务。
- 首日不同时开放商店、复杂战斗、社交、排行和全部状态系统。

### 4.2 重复捕获

- 首次：解锁图鉴、较强揭晓反馈。
- 重复：研究点、亲密度、外观碎片或任务进度。
- 高价值重复有保留/对比功能，普通重复可安全转化。
- 禁止纯客户端分解或发奖，避免资源作弊。

### 4.3 战斗

- 先做 PvE，再做 PvP。
- 战斗使用收藏动物的真实属性，终局服务端签名。
- 策略选择要改变风险/收益，而不是三个固定扣血按钮。
- 天气和状态只保留玩家能理解、能应对的影响，避免一周恶劣天气造成持续惩罚。

### 4.4 地图与城市

- 发现点是“观察建议”，不是鼓励追逐真实动物的精确定位。
- 展示大致区域、距离和精度，不展示其他玩家的精确捕获坐标。
- 城市收藏目标可提供长期动机，但不鼓励危险跨城或未成年人夜间出行。

### 4.5 经济

- 每个资源必须有来源、消耗和上限。
- 签到是回流辅助，不应成为最大金币来源。
- 道具增加策略或便利，不直接出售稀有度和战斗胜利。
- 每次配置变更跑 30/90 天蒙特卡洛，检查通胀、卡死和付费压力。

## 5. 识别质量与公平性

### 5.1 识别门槛

- 只承诺猫/狗/鹅；其他动物返回 unsupported/unknown。
- 阈值由黄金集校准，不为“看起来更容易成功”而随意降低鹅阈值。
- UI 显示置信度区间和可执行建议，不把模型猜测写成事实。

### 5.2 稀有度原则

- 不应把现实品种价格或社会偏见直接转化为战斗强度。
- 拍摄质量可影响展示、研究值或有限属性，但不能鼓励玩家逼近或惊扰动物。
- 核心 stats 使用确定性、版本化、可回放的服务端算法。
- 模型只辅助识别和叙事，不能让重试变成抽卡。

### 5.3 纠错闭环

- 玩家纠错进入匿名质量数据，但必须单独征得样本使用同意。
- 模型/Prompt/阈值每次变更都有离线基线和线上 canary。
- 监控 unknown、no-animal、纠正、重扫和放弃率，而不只看 2xx。

## 6. 安全、隐私与动物福利

- 不追逐、不投喂、不触摸、不惊扰动物。
- 排除道路、水边、施工区、私人区域和危险时段。
- 极端天气、车辆速度、低精度和低电量时暂停户外玩法。
- 上传前移除 EXIF，优先裁剪动物，遮挡人脸/车牌。
- 精确位置默认不保存；确有需要时单独 scope、短 TTL 和物理清理。
- 未成年人默认关闭精确位置分享、陌生社交和夜间户外任务。
- 商业化、排行榜和 PvP 上线前完成专门安全与公平评审。

## 7. 性能与成本预算

建议先设可测预算，再优化：

| 指标 | MVP 建议预算 |
|---|---|
| Detect P50 / P95 | <= 1.0s / <= 2.5s，先求稳定，再逐步压缩 |
| Analyze+Value P95 | <= 8s，并显示可取消进度 |
| 单次成功捕获 detect 次数 | 手动扫描 MVP <= 3 |
| 上传图片长边 | <= 1280，移除元数据 |
| 单次流程前台相机时长 | <= 5 分钟 |
| 核心流程错误率 | < 1%，且无静默数据丢失 |
| 同步队列 P95 年龄 | 在线时 <= 30s |
| 前端 JS gzip | 建立当前基线后设置硬预算 |

设计文档原有 300ms 端到端目标对云端 VLM + 移动网络过于激进。建议先用真实数据建立基线，再设分阶段目标，避免团队通过 Mock 达标。

## 8. 上线 Gate

任何公开测试前至少满足：

- 生产入口全链路 E2E 绿色。
- staging 与 production 数据和 Secret 完全隔离。
- 前后端镜像使用不可变 digest。
- OpenAPI 与 Runtime 无漂移。
- 商业化关闭或真实验签完成。
- 数据导出/删除、备份恢复、回滚演练通过。
- 识别黄金集达到约定阈值；unknown 不映射为支持物种。
- 核心流程没有无限奖励、重复扣费或静默同步失败。
- iOS/Android 真机完成权限、相机、后台、弱网、PWA 升级和无障碍矩阵。
